## 1. Сопоставление аудита с текущим состоянием проекта

Ниже — что из `audit.md` **уже в значительной степени покрыто** кодом/доками (не дублировать работу «с нуля», а при необходимости **расширять**):

| Тема audit | Статус в проекте (по TODO.md) | Комментарий |
|------------|-------------------------------|-------------|
| Rate limits, API key, WS Origin | ✅ P0–P1 | См. `SECURITY.md`, nginx-шаблон; в плане — polish UI предупреждений и согласование copy. |
| Upload: MIME + сигнатуры | ✅ P1 upload hardening | Фокус аудита SEC-1 про validation — вести как регрессионный чеклист и донастройку лимитов. |
| PostgreSQL, backup, мультипользователь | ✅ P2 | DB-1 в audit — это **аудит совместимости и тюнинга**, не «внедрить Postgres». |
| RAG пилот | ✅ P2.3 | RAG-1 в audit — качество retrieval, не включение RAG. |
| Reconnect / resume генерации | ✅ §20.1, P1 job queue | UX-2 в audit просит **видимый** reconnect — это главный пробел. |
| Structured logging / ErrorCode | ✅ частично P1.6 | BE-1 просит JSON-поля и `error_id` в WS — возможное расширение. |

**Вывод:** audit в первую очередь применим как **план продуктового и UX-коммита** и как **доработка observability/reliability**, а не как замена уже закрытых треков безопасности/платформы.

## 2. Идеи улучшения и расширения (сверх текста audit)

Связаны с архитектурой из [TODO.md §9.5.4](TODO.md#954-ограничения-и-отличия-от-идеала) и продуктовым ощущением:

1. **Качество ответа после img2img (опциональный v2-раунд):** после успешного ingest опционально подмешивать в контекст LLM `image_url` результата для финальной оценки — только за флагом и с лимитом токенов (не ломать «без base64 в контексте» принцип — URL как сейчас).
2. **Объяснимость RAG в UI:** блок «Ответ основан на…» (имена документов/фрагменты), если бэкенд уже отдаёт scores/sources — связать с `audit` RAG-1 «Context Explainability».
3. **Единый словарь статусов** в UI, согласованный с серверными `phase` / `tool_start` — не новые произвольные строки на клиенте, а маппинг из одного конфига (даже статического JSON или константы в `chat.js`), чтобы русскоязычные сообщения были консистентны.
4. **Motion и a11y:** любые анимации через `prefers-reduced-motion`; фокус-ловушки в модалках — профессиональный минимум.
5. **`reasoning_delta` в UI:** в протоколе уже есть ([TODO.md §1.5](TODO.md#15-модель-данных-sqlalchemy-20-async)) — аккуратный collapsible блок для совместимых моделей без шума в основном тексте.

## Фаза UX-1 — Design System Foundation

**Приоритет:** критический  
**Связь с audit:** [§Phase UX-1](audit.md)  
**Цель:** одна визуальная система (tokens, типографика, кнопки, карточки) для `chat`, `gallery`, `macros`, модалок, админ-панелей.

### Задачи

1. **Design tokens**
   - [x] Добавить `static/css/tokens.css` (или эквивалент): цвета surface/text/border, primary/danger/success, радиусы, шаг отступов, тени для light/dark.
   - [x] Подключить tokens **до** `chat.css`; постепенно заменить «разрозненные» переменные в `static/css/chat.css` без большого bang-rewrite одним коммитом (предпочтительно — по секциям: sidebar → composer → messages).
2. **Card / panel system**
   - [x] Базовый `.ui-card` + применение к composer; далее — settings/gallery/modals.
3. **Типографика**
   - [x] Токены `--type-*` в `tokens.css`, утилиты `.type-title`, `.type-meta`, `.type-hint` (placeholder).
   - [ ] Полное применение уровней по всем экранам; согласование с `--font-size` в настройках.
4. **Кнопки и icon-buttons**
   - [x] Добавлены `.btn-danger`, `.btn-icon` (базовые состояния).
   - [x] Composer: attach, @macro, «Ещё», send/cancel на общих классах; focus-visible rings.

### Чеклист готовности UX-1

- [ ] В одной беседе нет смешения «старого» и «нового» радиуса/бордера между sidebar и центром (допускается поэтапно, но **без** визуального «две темы сразу» в одном экране).
- [ ] Light/Dark переключаются без артефактов (проверить границы, скроллбары, selection).
- [ ] Focus ring виден при навигации с клавиатуры (Send, скрепка, пункты меню). — частично: composer + icon-buttons

### Опорная точка (gate) UX-1

| Проверка | Как |
|----------|-----|
| Визуальная консистентность | Скриншоты 3 экранов: чат, галерея, macros — одна «семья» отступов и поверхностей. |
| Регресс CSS | Открыть длинный тред с картинками, таблицей в markdown, lightbox — нет поломки вёрстки. |
| Контраст | Базовая проверка WCAG для текста/ссылок в light и dark (ручной контраст-чекер). |

## Фаза UX-2 — Chat Experience Polish

**Приоритет:** критический  
**Связь с audit:** [§Phase UX-2](audit.md)  
**Цель:** «живой» чат: понятные состояния потока, скелеты, reconnect, аккуратный composer, действия над сообщениями, человеческие статусы tools.

### Задачи

1. **Модель состояний клиента**
   - [x] State machine: `data-chat-ui-state` на `body` + `_recomputeUiState()` (`connecting | ready | streaming | tool_running | reconnecting | offline`).
   - [x] Единый pill в настройках через `_applyUiStateConnPill`; статус в пузыре — через `_resolveProgressLabel`.
2. **Skeleton loading**
   - [x] Скелет при первой загрузке истории / смене беседы (`_showMessagesSkeleton` в `loadMessages`).
   - [x] Скелет для поиска (`_showConvSearchSkeleton`) и галереи (`_showGallerySkeleton`).
3. **Reconnect UX (`static/js/chat.js`)**
   - [x] Лимит попыток `MAX_RECONNECTS`, backoff; баннер «Нет соединения» с кнопкой «Повторить».
   - [x] `_syncAfterReconnect()` — REST sync + refresh streaming bubble при восстановлении WS.
4. **Composer redesign**
   - [x] Зоны: вложения → ввод → toolbar (attach, @macro) → меню «Ещё» (каталог @alias, RAG) → send/cancel.
   - [x] Второстепенные контролы в `#composer-tools-menu`; CSS + JS toggle, focus rings.
5. **Message actions (hover / kebab)**
   - [x] Kebab-меню с подписями: копировать, редактировать, перегенерировать, удалить; скрытие на черновиках/во время стрима.
   - [ ] Новые действия (**pin**, **quote**, **retry**) — отложено до решения по модели данных (см. `MESSAGE_ACTION_DEFS` в `chat.js`).
6. **Assistant / tool статусы**
   - [x] `TOOL_USER_LABELS` + `PROGRESS_STAGE_LABELS` (русские подписи для tools и этапов).
   - [x] `_resolveProgressLabel()` — без сырых `snake_case` имён tools в UI.

### Чеклист готовности UX-2

- [ ] Обрыв сети во время стрима: пользователь видит понятный баннер; после восстановления — корректный финал или возможность обновить тред.
- [ ] F5 во время SD: сохраняется текущее поведение resume ([TODO.md §10.7](TODO.md#107-resume-генерации-после-f5)) без регресса дубля картинок.
- [ ] Send блокируется при активной генерации; textarea остаётся доступной как сейчас ([TODO.md §20.3](TODO.md#203-ux-чата-итерации)).

### Опорная точка (gate) UX-2

| Сценарий | Ожидание |
|----------|----------|
| Стрим текста | Индикация «thinking» до первого `text_delta`; затем плавное появление текста без мигания статусов. |
| `generate_image` | Статус «Генерация изображения…»; по `image` — превью в сетке; по `done` — стабильное состояние. |
| Reconnect | Искусственный `offline` в DevTools → баннер → online → продолжение без полной перезагрузки страницы. |
| Regression | `pytest -q`; ручной прогон img2img + PDF + RAG toggle. |
